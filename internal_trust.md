# Internal Trust for Platform Services

## Summary — What Gets Configured

This document describes the setup of internal trust between operators running on multiple OpenShift clusters and a centralized platform service. The pattern uses opaque Projected Service Account Tokens (PSATs) with cross-cluster token review to authenticate and authorize operator-to-service communication.

All configuration is **pure Kubernetes objects** — no external systems (F5, DNS, Azure) are involved. Cross-cluster Secret propagation is handled entirely by ACM using `ManagedClusterView`, hub templates, and `object-templates-raw`.

### On each operator cluster (ACM Policies 1–4, automatic)


| #   | Resource                                         | Purpose                                                                     |
| --- | ------------------------------------------------ | --------------------------------------------------------------------------- |
| 1   | **Namespace** (`platform-automation`)            | Dedicated namespace for operator + token-review infrastructure              |
| 2   | **ServiceAccount** (`token-review-sa`)           | SA used by the central service to validate tokens from this cluster         |
| 3   | **ClusterRoleBinding** → `system:auth-delegator` | Grants TokenReview/SubjectAccessReview permissions                          |
| 4   | **Secret** (`token-review-sa-token`, long-lived) | Auto-populated token for the token-review SA                                |
| 5   | **ServiceAccount** (`dataclassification-agent`)  | SA for the operator itself                                                  |
| 6   | **ClusterRole + ClusterRoleBinding**             | Operator workload permissions (e.g., watch namespaces)                      |
| 7   | **Deployment** (with projected SA token)         | Operator mounts a short-lived PSAT (audience: `platform-services`, 1hr TTL) |


### On the ACM hub (Policy 5, automatic)


| #   | Resource                                          | Purpose                                                                                                |
| --- | ------------------------------------------------- | ------------------------------------------------------------------------------------------------------ |
| 1   | **ManagedClusterView** (one per operator cluster) | Reads the `token-review-sa-token` Secret from each operator cluster back to the hub's `.status.result` |


### On the service cluster (ACM Policies 6–9, automatic)


| #   | Resource                                                            | Purpose                                                                                         |
| --- | ------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------- |
| 1   | **Namespace** (`platform-services`)                                 | Dedicated namespace for the central service                                                     |
| 2   | **ServiceAccount**                                                  | SA for the platform integration service                                                         |
| 3   | **Deployment + Service**                                            | Central service that receives PSATs and performs TokenReview                                    |
| 4   | **Secret** (`review-token-<clustername>`, one per operator cluster) | Created by hub templates reading from ManagedClusterViews — contains the review token + CA cert |
| 5   | **ConfigMap** (`cluster-registry`)                                  | List of operator cluster names, auto-generated by hub templates                                 |
| 6   | **ConfigMap** (`token-sync-ready`)                                  | Feedback signal — all policies Compliant when present                                           |


### Cross-cluster token propagation (pure ACM, no Ansible)


| Component                             | Role                                                               |
| ------------------------------------- | ------------------------------------------------------------------ |
| **ManagedClusterView**                | Bridges Secret data from operator cluster to the ACM hub           |
| **Hub templates** (`{{hub ... hub}}`) | Reads ManagedClusterView `.status.result` during policy evaluation |
| `**object-templates-raw`**            | Enables `range` loops to dynamically iterate over clusters         |
| **Self-healing**                      | New operator clusters are picked up on the next evaluation cycle   |


### Runtime flow (automatic, after setup)


| Step | What happens                                                                                           |
| ---- | ------------------------------------------------------------------------------------------------------ |
| 1    | Operator on Cluster A fetches a local opaque PSAT from its API server                                  |
| 2    | Operator calls the central Service (on Cluster B) presenting the opaque PSAT                           |
| 3    | Service iterates over its configured cluster tokens, attempting TokenReview against each cluster's API |
| 4    | The correct cluster's API returns user info (ServiceAccount name, namespace)                           |
| 5    | Service uses the ServiceAccount identity + cluster name to authorize the API call                      |


### Key design properties

- **Short-lived access tokens** — the actual workload PSATs are short-lived and secure
- **Long-lived review tokens** — only the TokenReview tokens are long-lived (used server-side, not transmitted to clients)
- **No shared secrets** — each cluster's API is the authority for its own tokens
- **Authorization is identity-based** — ServiceAccount name + namespace + cluster name determines access
- **Pure Kubernetes/ACM** — no Ansible, no external automation platforms needed

Setup of internal trust for platform services

Operators across multiple clusters should be able to call a central automation service with PSAT auth.

Example: on each cluster, there exists a dataclassification-agent-operator, that runs in context of the platform automation team.
The operator must keep a central service updated with information obtained when a new namespace is created.
The tokens are opaque and need to be reviewed on the API that issued them. This requires a service account token with token review access.

Once the user info is available, the service account name, namespace and cluster is combined to authorize the use of the service API.

This means that there's an initial setup phase that is currently handled manually on every addition of a new cluster:

1. On cluster with operator:
  a. generate long-lived token with access to token-review
2. On cluster with integration services:
  a. add token as a secret
  b. add token secret to configuration for platform services to mount it
  c. add token mount to services (includes service rollout)

This solution is a little work to set up, but wins out because the actual access tokens are short lived and secure.
